# BPF Assembly and JIT

## Stopping Shellcode Without Breaking the JIT

The JVM's JIT compiler is the most obvious obstacle to strict memory protection. The JIT must call `mmap` with `PROT_EXEC` to allocate executable memory for optimized native code. A naive Seccomp filter blocking all `PROT_EXEC` will crash the JVM immediately.

`mazewall` solves this with **BPF argument inspection**. The BPF filter doesn't just check the syscall number — it checks the `prot` argument to `mmap` (and `mprotect`). Specifically, it loads the lower 32 bits of `prot` (located at `args[2]` in `struct seccomp_data`) and tests whether the `PROT_EXEC` bit (0x4) is set. 

Here is the exact logical flow of the BPF bytecode instruction sequence generated by the library:

```
; In Classic BPF, jt and jf are INSTRUCTION SKIP COUNTS, not jump labels.
; jt=N means "if condition TRUE, skip N instructions forward".
; jf=N means "if condition FALSE, skip N instructions forward".

; 1. Load the system call number into the accumulator
BPF_LD offset=0        ; ACC = seccomp_data.nr  (syscall number)

; 2. If not mmap (syscall 9 on x86_64), skip 4 instructions over the mmap block
BPF_JEQ #9  jt=0 jf=4 ; ACC==9? true: jt=0 (fall into mmap check); false: jf=4 (skip block)

; 3. Load the prot argument (args[2] at byte offset 32 in seccomp_data)
BPF_LD offset=32       ; ACC = seccomp_data.args[2] lower 32 bits = prot

; 4. Test PROT_EXEC bit (0x04): opcode BPF_JSET = BPF_JMP|0x40|BPF_K
BPF_JSET #0x04  jt=0 jf=1
;   (ACC & 0x04) != 0 → PROT_EXEC IS set: jt=0 → fall through to RET_DENY immediately
;   (ACC & 0x04) == 0 → PROT_EXEC not set: jf=1 → skip over RET_DENY, continue

BPF_RET SECCOMP_RET_ERRNO|EPERM   ; deny: return EPERM to caller

BPF_LD offset=0        ; restore ACC = syscall number for subsequent filter checks
; (identical 5-instruction block follows for mprotect)
```

Because this filter is applied only to the contained worker threads, the JIT threads running on the same JVM continue operating without restriction. We've surgically neutralized shellcode execution — thread-scoped, zero JVM stability impact.

> **32-bit truncation note:** BPF operates on 32-bit words. The filter loads the lower 32 bits of `prot`. Since the Linux kernel casts the `prot` argument to `unsigned long` but only honors the standard lower bits defined in POSIX, this truncation is architecturally correct and matches kernel behavior.
